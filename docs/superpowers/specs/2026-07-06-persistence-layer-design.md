# Persistence Layer Design — Kanban Backend

**Date:** 2026-07-06
**Status:** Approved
**Scope:** JPA entities, Liquibase schema, and Spring Data repositories for the Kanban ticket tracker — including the **complete User data model** (users + email-verification tokens). Authentication *logic* (Argon2id hashing, login/logout, sessions/bearer tokens, SMTP delivery, and the verification/resend flow) is **deferred** to a later phase; only the persistent structures for it are created now.

## 1. Context

The backend is Spring Boot 4.1.0 on Java 25, using Spring Data JPA + Liquibase against PostgreSQL 18, with Lombok. Liquibase owns schema management (`spring.jpa.hibernate.ddl-auto=validate`), so **entity mappings must match the Liquibase-generated schema exactly** or the application fails to start. The master changelog (`db/changelog/db.changelog-master.yaml`) already uses `includeAll` on `db/changelog/changes/`, so new change files are dropped into that directory and applied in alphabetical order.

Per the requirements, a fresh database must contain schema and migration metadata only — **no seed or sample data**.

### Decisions taken during brainstorming

- **Primary keys:** `uuid`, application-generated. Avoids ID enumeration in API URLs; the spec lists UUIDs first.
- **User entity:** modeled **completely now** — the `users` table carries `password_hash`, `email_verified`, and `display_name`, and a separate `email_verification_tokens` table holds the mandatory single-use/expiring tokens. This avoids a schema-churn migration when auth logic lands. The columns simply go unused until that phase.
- **User foreign keys** (`ticket.created_by_id`, `comment.author_id`): **NOT NULL** — matches the spec's "required." The integration test and the future auth-backed API both supply a user, so nothing is blocked.
- **Repositories:** included in this work — they are the persistence tier's interface.

## 2. Package structure

Feature-based packages, one per aggregate, with **unidirectional** relations (`@ManyToOne` on the child side only — no `@OneToMany` collections):

```
com.berzantas.kanban
├── user     → User, EmailVerificationToken, UserRepository, EmailVerificationTokenRepository
├── team     → Team,   TeamRepository
├── epic     → Epic,   EpicRepository
├── ticket   → Ticket, TicketType, TicketState, TicketRepository
└── comment  → Comment, CommentRepository
```

Lists and boards are fetched via repository queries, not via entity collection navigation. This keeps each entity small, focused, and independently testable.

## 3. Entities

All primary keys are `uuid`, generated by the application (Hibernate UUID generation). All timestamps are `timestamptz` stored in UTC.

### User (`users`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| email | varchar(320) | NOT NULL; unique case-insensitive (index on `LOWER(email)`) |
| password_hash | varchar(255) | NOT NULL; stores the encoded Argon2id hash (e.g. `$argon2id$...`). Never a plaintext password. |
| email_verified | boolean | NOT NULL; default `false` |
| display_name | varchar(255) | NOT NULL |
| created_at | timestamptz | NOT NULL |
| modified_at | timestamptz | NOT NULL |

Email is trimmed and compared case-insensitively at the service layer; the DB unique index on `LOWER(email)` is the backstop. `password_hash` and `email_verified` are populated once auth logic is implemented; the persistence layer just provides the columns.

### EmailVerificationToken (`email_verification_tokens`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| user_id | uuid | NOT NULL; FK → users(id) ON DELETE CASCADE |
| token | varchar(255) | NOT NULL; unique |
| expires_at | timestamptz | NOT NULL (24h after issue) |
| consumed_at | timestamptz | nullable; set when the single-use token is redeemed (`NULL` = unused) |
| created_at | timestamptz | NOT NULL |

Single-use (`consumed_at`) and expiry (`expires_at`) are enforced by the auth-phase service. "Issuing a new token invalidates earlier unused tokens" is likewise service logic (mark prior unconsumed tokens as consumed); the schema supports it. Tokens cascade-delete with their user.

### Team (`teams`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| name | varchar(255) | NOT NULL; unique case-insensitive (index on `LOWER(name)`) |
| created_at | timestamptz | NOT NULL |
| modified_at | timestamptz | NOT NULL |

Name must be non-empty after trimming (service-layer validation).

### Epic (`epics`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| team_id | uuid | NOT NULL; FK → teams(id) ON DELETE RESTRICT |
| title | varchar(255) | NOT NULL (non-empty after trimming) |
| description | text | nullable |
| created_at | timestamptz | NOT NULL |
| modified_at | timestamptz | NOT NULL |

`team_id` is set at creation and never changed. A unique constraint on `(id, team_id)` exists to serve as the target of the ticket composite FK (see §5).

### Ticket (`tickets`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| team_id | uuid | NOT NULL; FK → teams(id) ON DELETE RESTRICT |
| epic_id | uuid | nullable; part of composite FK → epics(id, team_id) ON DELETE RESTRICT |
| type | varchar(16) | NOT NULL; CHECK IN ('BUG','FEATURE','FIX') |
| state | varchar(32) | NOT NULL; CHECK IN ('NEW','READY_FOR_IMPLEMENTATION','IN_PROGRESS','READY_FOR_ACCEPTANCE','DONE') |
| title | varchar(255) | NOT NULL (non-empty after trimming) |
| body | text | NOT NULL (non-empty) |
| created_by_id | uuid | NOT NULL; FK → users(id) |
| created_at | timestamptz | NOT NULL |
| modified_at | timestamptz | NOT NULL |

### Comment (`comments`)
| Column | Type | Constraints |
|---|---|---|
| id | uuid | PK |
| ticket_id | uuid | NOT NULL; FK → tickets(id) ON DELETE CASCADE |
| author_id | uuid | NOT NULL; FK → users(id) |
| body | text | NOT NULL (non-empty) |
| created_at | timestamptz | NOT NULL |

Comments are immutable (no `modified_at`). Displayed oldest-first.

### Enums

Java enums are idiomatic uppercase and persisted via `@Enumerated(EnumType.STRING)`:

- `TicketType`: `BUG`, `FEATURE`, `FIX`
- `TicketState`: `NEW`, `READY_FOR_IMPLEMENTATION`, `IN_PROGRESS`, `READY_FOR_ACCEPTANCE`, `DONE`

The stored strings are guarded by DB `CHECK` constraints. The future API layer maps these to the spec's lowercase canonical JSON values (`bug`, `ready_for_implementation`, …) via DTOs — the persistence layer does not expose JSON.

### Timestamps

`created_at` and `modified_at` are set on insert via Hibernate `@CreationTimestamp`. The requirement that "saving unchanged values must not advance `modified_at`" is service-layer logic added later; the persistence layer only provides the columns.

## 4. Relationships (summary)

Every relationship is child → parent, with the FK column on the child.

| From (child) | → To (parent) | Cardinality | FK column | Nullable? | On delete of parent |
|---|---|---|---|---|---|
| Epic | Team | many → one | `team_id` | No | RESTRICT (→ 409) |
| Ticket | Team | many → one | `team_id` | No | RESTRICT (→ 409) |
| Ticket | Epic | many → one | `epic_id` | Yes | RESTRICT (→ 409) |
| Ticket | User | many → one | `created_by_id` | No | NO ACTION |
| Comment | Ticket | many → one | `ticket_id` | No | **CASCADE** |
| Comment | User | many → one | `author_id` | No | NO ACTION |
| EmailVerificationToken | User | many → one | `user_id` | No | **CASCADE** |

## 5. Referential integrity & the same-team rule

Database constraints act as the backstop to service-layer validation:

- **Delete guards** (Team↔Epic, Team↔Ticket, Epic↔Ticket) use `ON DELETE RESTRICT`, so a violating delete raises a DB error that the service maps to **HTTP 409 Conflict**.
- **Ticket → Comment** uses `ON DELETE CASCADE`, so deleting a ticket removes its comments in one operation.
- **Same-team epic rule:** a ticket and its epic must belong to the same team. Enforced at the DB level by:
  - a unique constraint on `epics(id, team_id)`, and
  - a **composite foreign key** `tickets(epic_id, team_id) → epics(id, team_id)`.

  With Postgres `MATCH SIMPLE` (default), a `NULL` `epic_id` skips the check (the epic is optional), but any non-null `epic_id` forces the referenced epic to share the ticket's `team_id`. The service layer also validates this, but the database guarantees it.

## 6. Liquibase changelogs

YAML files (matching the existing changelog format) in `db/changelog/changes/`, numbered so the alphabetical `includeAll` order is FK-safe:

1. `001-create-users.yaml`
2. `002-create-email-verification-tokens.yaml`
3. `003-create-teams.yaml`
4. `004-create-epics.yaml` — includes unique constraint on `(id, team_id)`
5. `005-create-tickets.yaml` — includes composite FK to `epics(id, team_id)`, CHECK constraints, FK indexes
6. `006-create-comments.yaml`

Each change set includes rollback statements. Case-insensitive uniqueness (`teams.name`, `users.email`) is created via raw-SQL `CREATE UNIQUE INDEX … (LOWER(col))` change sets (with matching `DROP INDEX` rollbacks), since functional indexes are not portably expressible in Liquibase's structured YAML.

**FK-supporting indexes** (Postgres does not auto-index foreign keys):
- `email_verification_tokens(user_id)`
- `epics(team_id)`
- `tickets(team_id)`, `tickets(epic_id)`, `tickets(created_by_id)`
- `comments(ticket_id)`, `comments(author_id)`

No seed data is loaded — a fresh database contains schema + Liquibase metadata only.

## 7. Repositories

Spring Data JPA interfaces (`JpaRepository<T, UUID>`) with the query methods the integrity rules require:

- `UserRepository`: `existsByEmailIgnoreCase(String)`, `findByEmailIgnoreCase(String)`
- `EmailVerificationTokenRepository`: `findByToken(String)`, `findByUserIdAndConsumedAtIsNull(UUID)` (for invalidating earlier unused tokens)
- `TeamRepository`: `existsByNameIgnoreCase(String)`, `findByNameIgnoreCase(String)`
- `EpicRepository`: `findByTeamId(UUID)`, `existsByTeamId(UUID)` (team delete-guard)
- `TicketRepository`: `findByTeamId(UUID)`, `existsByTeamId(UUID)` (team guard), `existsByEpicId(UUID)` (epic guard)
- `CommentRepository`: `findByTicketIdOrderByCreatedAtAsc(UUID)`

Complex board filtering (type/epic/title substring search, AND-combined) is deferred to the API/service phase.

## 8. Testing

Add **Testcontainers** (`org.testcontainers:postgresql` + `org.testcontainers:junit-jupiter`, test scope) to `pom.xml`. H2 cannot model `timestamptz`, `LOWER()` functional indexes, or composite foreign keys, so the schema must be tested against real PostgreSQL.

An integration test (`@SpringBootTest` with a Testcontainers Postgres) verifies:

1. Liquibase migrations apply cleanly **and** Hibernate `validate` passes (entity ↔ schema alignment) — this alone catches most mapping drift.
2. Case-insensitive uniqueness on `users.email` and `teams.name`.
3. Ticket → Comment cascade delete.
4. Rejection of a ticket whose `epic_id` belongs to a different team (composite-FK enforcement).

This satisfies the requirement for an automated test covering at least one backend business flow.

## 9. Out of scope (this phase)

- Authentication *logic*: Argon2id hashing/verification, login/logout, session or bearer-token issuance. (The `password_hash` column exists but nothing populates it yet.)
- Email-verification *flow*: SMTP delivery, token generation/redemption, 24h expiry and single-use enforcement, and resend/invalidation logic. (The `email_verification_tokens` table exists but no logic operates on it yet.)
- Service and API/controller layers, DTOs, and enum JSON case-mapping.
- Board filtering/search query implementation.
